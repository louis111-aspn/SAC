1. DCL vs. ABAP Authority-Check — 두 권한 메커니즘의 차이

S/4HANA에서 CDS View 기반 권한 제어는 두 가지 메커니즘이 존재합니다. DCL(Data Control Language)은 CDS 레이어에서 행(Row) 수준 필터를 선언적으로 적용하고, ABAP Authority-Check는 프로그램 레이어에서 명령적으로 검사합니다. Analytics Query CDS View 환경에서는 DCL이 표준입니다.

항목 ABAP Authority-Check CDS DCL (DEFINE ROLE)
적용 위치 ABAP 프로그램 코드 내 CDS View에 선언적으로 연결 (.dcl 파일)
작동 방식 명령적(Imperative) — 코드 실행 시 검사 선언적(Declarative) — DB 레벨 행 필터로 자동 적용
OData / SAC 환경 OData 서비스에 적용 안 됨 OData·SAC 모든 접근 경로에 자동 적용
권한 오브젝트 PFCG 권한 오브젝트 직접 참조 DCL 내 ASPECT pfcg_auth로 PFCG 오브젝트 연동
상속 없음 — View마다 개별 구현 하위 CDS View로 자동 전파 (상속 범위 제어 가능)
관리 편의성 코드 변경 필요 — 운영 리스크 PFCG Role만 변경하면 즉시 반영
Analytics Query 적용 불가 (프로그램 없음) ✓ 표준 — CDS View에 직접 적용

Dialog User(SAC 분석가), Communication User(SAC Technical User). 각각 PFCG Role 할당. SAC 접속 시 할당된 Role의 권한 필터가 DCL을 통해 자동 적용.

@AccessControl.authorizationCheck: #NOT_REQUIRED — 사용 판단 기준

Tab 02에서도 언급했지만 재강조합니다. 공개 코드 테이블·통화 코드·단위 마스터처럼 모든 사용자에게 동일하게 노출되어도 무방한 공용 참조 데이터에는 #NOT_REQUIRED가 정상적으로 사용됩니다.

그러나 재무·인사·영업 등 업무 데이터에는 사용하지 않습니다. SAC Live Model에 연결된 Query View·Cube View에 이 설정이 남아 있으면 권한 없는 사용자에게 ASPN Group 전체 재무 데이터가 노출됩니다. 개발·디버깅 완료 후 반드시 #CHECK로 전환하고 DCL을 작성합니다.

2. DEFINE ROLE 구문 — Access Control Object

DCL은 ADT에서 .dcl 파일로 작성합니다. New → Other ABAP Repository Objects → Core Data Services → Access Control을 선택합니다. 파일명은 보호 대상 CDS View와 동일하게 짓는 것이 표준입니다.

ABAP DCL ZASPN_C_FinancialCube — Access Control 전체 해부
1
@MappingRole: true
자동 매핑 선언. @MappingRole: true를 선언하면 DCL 파일명과 동일한 CDS View에 자동 연결됩니다. CDS View 코드를 수정하지 않아도 이 DCL이 적용됩니다.
2–4
define role ZASPN_C_FinancialCube { grant select on ZASPN_C_FinancialCube where (
ROLE 선언. Role 이름은 보호 대상 CDS View와 동일하게 선언하는 것이 표준. grant select on: SELECT(읽기) 권한을 조건부로 허용.
5–12
/* ── PFCG 권한 오브젝트 연동 ── */ /* 회사 코드 권한: F_BKPF_BUK */ CompanyCode = aspect pfcg_auth( 'F_BKPF_BUK', /* 권한 오브젝트 이름 */ 'BUKRS', /* 권한 오브젝트 필드 */ 'ACTVT' = '03' /* 활동 코드: 03 = Display */ )
PFCG 권한 오브젝트 연동. ASPECT pfcg_auth: 지정한 권한 오브젝트의 허용 값으로 행을 필터링. F_BKPF_BUK: 회계 전표 회사 코드 권한 오브젝트. ACTVT = '03': Display(조회) 활동.
13–20
and /* 관리 회계 영역 권한: K_CCA */ ControllingArea = aspect pfcg_auth( 'K_CCA', /* CO 영역 권한 오브젝트 */ 'KOKRS', /* Controlling Area 필드 */ 'ACTVT' = '03' )
복수 권한 오브젝트 AND 조건. 두 조건이 모두 충족되어야 데이터에 접근 가능. ASPN Group에서 Korea 컨트롤링 영역(KR01)과 Europe 컨트롤링 영역(EU01)에 권한이 없으면 해당 데이터가 필터됩니다.
21
); }
Role 선언 종료. 세미콜론으로 WHERE 조건을 닫고 Role을 닫습니다.

3. DCL Condition 유형

DCL WHERE 조건에는 세 가지 유형이 있습니다. 실제 프로젝트에서는 ASPECT pfcg_auth가 주력이지만, 정적 필터와 태그 기반 제어도 상황에 따라 활용합니다.

가장 많이 사용
ASPECT pfcg_auth
PFCG 권한 오브젝트 연동
사용자에게 할당된 PFCG Role의 권한 오브젝트 값으로 행을 필터링합니다. 사용자별로 다른 회사 코드·이익센터에 접근하는 동적 필터에 사용합니다.

ASPN 예시: Korea 담당자 → CompanyCode = '2000'만 허용, Europe 담당자 → '3000'만 허용. PFCG Role 변경만으로 즉시 적용.
정적 필터
WHERE 조건 (리터럴)
aspect literal condition
하드코딩된 조건으로 모든 사용자에게 동일하게 적용됩니다. 특정 회계연도 이후 데이터만 노출하거나, 삭제 표시 레코드를 항상 제외하는 등 전역 고정 필터에 사용합니다.

FiscalYear >= '2020'처럼 선언하면 모든 사용자에게 적용됩니다.
태그 기반
ASPECT entity_tag
태그 속성 기반 제어
CDS View의 특정 필드에 태그를 부여하여 권한을 제어하는 방식입니다. 표준 사용 빈도가 낮고 복잡한 시나리오에 활용합니다.

주로 외부 Authorization Provider와 연동하거나, 사용자 속성(부서, 지역) 기반 동적 필터에 활용합니다.

ASPN Group — 복수 조건 DCL 완성 예시

ABAP DCL ZASPN_C_FinancialCube — PFCG + 정적 필터 복합 조건 
@MappingRole: true

define role ZASPN_C_FinancialCube {
  grant select on ZASPN_C_FinancialCube
    where (

      /* ① 회사 코드 권한 (PFCG) */
      CompanyCode = aspect pfcg_auth(
          'F_BKPF_BUK', 'BUKRS', 'ACTVT' = '03'
      )

      and

      /* ② 관리 회계 영역 권한 (PFCG) */
      ControllingArea = aspect pfcg_auth(
          'K_CCA', 'KOKRS', 'ACTVT' = '03'
      )

      and

      /* ③ 정적 필터: 2020년 이후 데이터만 노출 */
      FiscalYear >= '2020'

      and

      /* ④ 원장: Leading Ledger만 노출 (운영 정책) */
      Ledger = '0L'
    );
}
ASPN Group 권한 설계 시나리오

ASPN Holdings CFO: F_BKPF_BUK에 CompanyCode 1000·2000·3000 전체 허용 → 전 법인 재무 조회
ASPN Korea CFO: F_BKPF_BUK에 CompanyCode 2000만 허용 → Korea 데이터만 조회
ASPN Europe Controller: F_BKPF_BUK에 CompanyCode 3000만 허용 → Europe 데이터만 조회

DCL의 PFCG 연동으로 사용자별로 자동 필터가 적용되며, SAC에서도 동일하게 동작합니다.

4. CDS View에서 자주 사용하는 PFCG 권한 오브젝트

권한 오브젝트 주요 필드 제어 대상 ASPN 활용
F_BKPF_BUK BUKRS, ACTVT 회계 전표 — 회사 코드 ASPN Holdings/Korea/Europe 법인별 재무 데이터 분리
F_BKPF_KOA KOART, ACTVT 회계 전표 — 계정 유형 (G/L, 자산, 채권 등) G/L 계정(S) 접근만 허용하는 Analytics 사용자 제한
K_CCA KOKRS, ACTVT 원가 센터 — 관리 회계 영역 Korea CO 영역(KR01)과 Europe CO 영역(EU01) 분리 접근
K_PCA KOKRS, PRCTR, ACTVT 이익 센터 영업본부(PC_1010) 담당자에게 해당 이익센터만 노출
F_SKA1_BUK BUKRS, ACTVT G/L 계정 마스터 — 회사 코드 Dimension View(GL 계정) DCL에 적용
G_SEGM SEGMENT, ACTVT 세그먼트 ASPN Europe의 사업 세그먼트별 분리 접근 제어
ACTVT (활동 코드) 표준값

'01': Create(생성)  |  '02': Change(변경)  |  '03': Display(조회)  |  '06': Delete(삭제)  |  '*': 전체 활동

Analytics Query CDS View는 읽기 전용이므로 DCL에서 ACTVT = '03'을 사용합니다. '01'이나 '02'를 조건으로 넣으면 조회 전용 사용자의 데이터가 필터될 수 있습니다.

5. Access Control 상속 — 하위 View로의 전파

CDS View A가 CDS View B를 소스로 참조할 때, B에 설정된 DCL이 A를 통한 접근에도 적용됩니다. 이 상속(Inheritance) 동작을 정확히 이해해야 의도치 않은 데이터 누출이나 차단을 막을 수 있습니다.

상속 결과: SAC에서 ZASPN_C_PLQuery로 데이터를 조회하면 Query View DCL + Cube View DCL + I_JournalEntryItem 표준 DCL이 모두 AND 조건으로 적용됩니다. 가장 엄격한 조건이 우선합니다.

Access Control 상속 동작 — #CHECK vs. #NOT_REQUIRED

ABAP CDS @AccessControl.authorizationCheck 값별 상속 동작 
// ── @AccessControl.authorizationCheck 옵션 ──────────────────────────

// ① #CHECK (표준 권장)
//    이 View에 DCL이 있으면 적용됨
//    소스 View(하위 View)의 DCL도 함께 AND 조건으로 전파 적용
@AccessControl.authorizationCheck: #CHECK

// ② #NOT_REQUIRED
//    이 View의 DCL을 완전히 우회 — 모든 행 노출
//    소스 View DCL도 전파되지 않음
//    업무 데이터(재무·인사·영업)에는 사용 지양. 공용 참조 데이터에는 정상 사용 가능
@AccessControl.authorizationCheck: #NOT_REQUIRED

// ③ #PRIVILEGED_ONLY
//    ABAP 코드에서 WITH PRIVILEGED ACCESS 키워드로 호출 시에만
//    DCL 우회 허용. 일반 OData·SAC 접근에는 DCL 정상 적용
//    내부 배치 처리·마이그레이션 프로그램에서 활용
@AccessControl.authorizationCheck: #PRIVILEGED_ONLY

// ── 상속 전파 원칙 ───────────────────────────────────────────────────
// • #CHECK 선언 시: 소스 View들의 DCL이 상위 View 조회에도 AND 조건으로 누적 적용
// • SAP 표준 I_JournalEntryItem의 DCL이 ZASPN_C_FinancialCube 조회 시에도 적용됨
// • 상속을 명시적으로 차단하는 CDS Annotation은 존재하지 않음
// • DCL 범위를 제한하려면 해당 소스 View를 별도 wrapper View로 분리하여 설계
상속 설계 원칙

① Cube View와 Query View 모두에 DCL을 작성합니다. Query View에만 DCL을 두면 Cube를 직접 참조하는 다른 Query에서 권한 우회가 가능합니다.
② Dimension View에도 DCL을 작성합니다. Value Help 조회 시에도 권한 필터가 적용되어야 합니다.
③ SAP 표준 Interface View(I_JournalEntryItem)는 SAP 표준 DCL이 이미 적용되어 있습니다. 커스텀 Cube View에서 추가 DCL을 작성하면 AND 조건으로 더 엄격하게 제어됩니다.

6. SAC Technical User (Communication User) 권한 설계

SAC가 S/4HANA에 Live Connection으로 접속할 때 사용하는 계정이 Communication User입니다. 이 계정의 권한 설계가 잘못되면 SAC 전체 사용자가 데이터를 못 보거나 반대로 모든 데이터가 노출되는 두 가지 위험이 동시에 존재합니다.

①

Communication User 생성 원칙

SU01에서 User Type을 Communication User로 생성합니다. Dialog User가 아닌 Communication User만 사용합니다. 비밀번호 만료가 없으며 대화형 로그인이 불가하여 보안이 강화됩니다. ASPN Group 예시: ZASPN_SAC_COMM

②

Technical User의 PFCG Role — 전체 데이터 접근 설계

Communication User는 SAC가 모든 사용자의 데이터를 대신 조회하는 구조입니다. 따라서 Communication User에게는 전체 회사 코드·영역에 대한 조회 권한을 부여합니다. 실제 사용자 데이터 필터링은 SAC의 사용자별 데이터 접근 제어로 처리합니다.

③

OData 서비스 접근 권한

Communication User에게 S_SERVICE 권한 오브젝트로 등록된 OData 서비스에 대한 접근 권한을 부여합니다. SRV_NAME에 Query View의 OData 서비스명을 명시합니다.

④

SAC 사용자별 데이터 접근 제어 — 별도 레이어

SAC 자체의 데이터 접근 모델(Data Access Control)로 개별 SAC 사용자가 볼 수 있는 데이터 범위를 제한합니다. Communication User는 전체 데이터를 가져오지만 SAC가 각 사용자의 권한에 맞게 필터링하여 표시합니다. 이 두 레이어를 혼동하면 보안 설계 오류가 발생합니다.

⑤

CDS DCL과 SAC 데이터 접근 제어의 관계

CDS DCL은 S/4HANA 레이어에서 동작합니다. Communication User의 PFCG Role에 의해 DCL이 적용됩니다. SAC 데이터 접근 제어는 SAC 레이어에서 별도로 동작합니다. 두 레이어가 모두 올바르게 설정되어야 종합적인 보안이 확보됩니다. 어느 한 레이어만 설정해서는 안 됩니다.

흔한 운영 실수 — Communication User에 제한된 권한 부여

Communication User에게 특정 회사 코드만 허용하면 해당 회사 코드의 SAC 사용자도 데이터를 볼 수 없게 됩니다. Communication User는 SAC 전체 사용자를 대표하므로 모든 접근 가능 데이터 범위의 권한을 보유해야 합니다. 개별 사용자 수준의 필터는 SAC 데이터 접근 제어에서 처리합니다.

권한 오브젝트필드허용 값목적
