코드를 짜는 것이 목표가 아닙니다

SAC Custom Widget 개발은 전통적으로 JavaScript 개발자의 영역이었습니다. 그러나 Claude를 개발 파트너로 활용하면 컨설턴트가 직접 위젯을 설계하고 완성할 수 있습니다. 이 교육과정은 그 협업 방법을 다룹니다.

❌ 기존 방식
  • JavaScript 개발자에게 요청 위임
  • 요구사항 전달 → 개발 → 수정 사이클 반복
  • 코드 내용을 이해하지 못한 채 납품
  • 오류 발생 시 개발자 재요청 필요
  • 컨설턴트가 결과물을 주도하기 어려움

✓ 이 교육과정의 방식
  • 컨설턴트가 직접 요구사항을 Claude에게 전달
  • Claude가 파일 생성 → 컨설턴트가 SAC에 등록·테스트
  • 오류 메시지를 Claude에게 그대로 전달하여 해결

기본 개념

SAC Custom Widget이란

SAC(SAP Analytics Cloud)는 기본 제공되는 표준 위젯(차트, 테이블, 필터 등) 외에, 개발자가 직접 만든 UI 컴포넌트를 스토리에 삽입할 수 있도록 Custom Widget 기능을 지원합니다. 기반 기술은 W3C 표준인 Web Component입니다.

표준 위젯으로 충분한 경우

Custom Widget 불필요

데이터 시각화, 테이블 표시, 기본 필터 적용, 표준 입력 셀 형태의 계획 입력

Custom Widget이 필요한 경우

이런 때 써야 합니다

계층형 메뉴 내비게이션, AI API 직접 연동, 완전히 커스텀한 입력 폼, 외부 데이터 실시간 호출 등 표준 기능의 범위를 벗어나는 UI

파일 구조

3개 파일로 구성됩니다

SAC Custom Widget은 반드시 3개의 파일로 구성되며, 각 파일의 역할이 명확히 분리되어 있습니다. 이 구조를 이해하는 것이 Claude에게 요구사항을 정확히 전달하는 출발점입니다.

JSON
widget.json 위젯의 신분증입니다. 위젯 이름, 고유 ID, 파라미터 정의, SAC가 인식해야 할 모든 메타데이터가 담깁니다. SAC는 이 파일을 먼저 읽어 위젯 등록 가능 여부를 판단합니다.
JS
main.js 위젯의 실제 동작과 화면을 담당합니다. 화면에 무엇을 표시할지(HTML), 어떻게 보일지(CSS), 클릭·입력 등 사용자 상호작용에 어떻게 반응할지(JavaScript)가 모두 이 파일 안에 있습니다.
JS
styling.js SAC 스토리 편집 화면 우측의 Styling Panel을 담당합니다. API Key, System Prompt, 최대 토큰 수 등 위젯 설정값을 편집할 수 있는 입력 폼을 제공합니다.
ZIP 패키징 규칙 3개 파일을 ZIP으로 묶을 때 반드시 루트에 직접 포함해야 합니다. my_widget/widget.json처럼 서브폴더 안에 넣으면 SAC가 인식하지 못합니다.
등록 절차

SAC에 위젯 등록하는 방법

1

ZIP 파일 준비

widget.json, main.js, styling.js 3개 파일을 서브폴더 없이 ZIP으로 압축합니다.

2

Custom Widgets 메뉴 진입

SAC 메인 메뉴 → Stories → Custom Widgets 탭으로 이동합니다.

3

widget.json 먼저 업로드

"+" 버튼 클릭 → widget.json 파일 선택. ZIP이 아닌 JSON 파일만 선택합니다.

4

ZIP 파일 업로드

JSON 업로드 후 나타나는 "Select File" 버튼 → 준비한 ZIP 파일 선택.

5

등록 확인 후 스토리에서 사용

Custom Widgets 목록에 위젯이 표시되면 완료. 이후 스토리 편집 화면의 위젯 패널에서 드래그하여 캔버스에 배치합니다.

재등록 시 주의 동일한 id의 위젯을 수정하려면 기존 위젯을 삭제한 후 재등록해야 합니다. 삭제 전 해당 위젯을 사용 중인 스토리 목록을 반드시 확인하십시오.

Step 1 · 요구사항 정의

좋은 요구사항 프롬프트 작성법

Claude에게 전달하는 초기 프롬프트의 품질이 개발 사이클 횟수를 결정합니다. 다음 4가지를 반드시 포함하십시오.

① UI 동작

화면이 어떻게 보이고 작동해야 하는가

레이아웃, 색상, 클릭·입력 시 동작 방식을 구체적으로 묘사합니다. "예쁘게"가 아니라 "좌측에 트리 메뉴, 클릭 시 오른쪽 영역 전환"처럼 서술합니다.

② 데이터 연동

어떤 데이터를 어떻게 받는가

SAC 스크립트에서 어떤 형태로 데이터를 전달할지 명시합니다. 예: "SAC 스크립트에서 문자열 형태로 JSON을 전달하는 setContext() 메서드가 필요합니다."

③ SAC 제약 명시

SAC 환경 제약사항을 반드시 포함

SAC Custom Widget은 일반 웹과 다른 제약이 있습니다. 이를 Claude에게 미리 알려야 오류를 줄일 수 있습니다. (다음 섹션 참고)

④ 파일 구조

3파일 구조임을 명시

"widget.json, main.js, styling.js 3개 파일을 각각 작성해 주세요"라고 명시합니다. 누락하면 Claude가 단일 파일로 작성할 수 있습니다.

Step 2 · Claude 프롬프트 구성

SAC 제약사항 — Claude에게 반드시 전달해야 할 내용

SAC Custom Widget 환경에는 일반 JavaScript와 다른 구조적 제약이 있습니다. 이를 초기 프롬프트에 포함하거나, 오류 발생 시 함께 전달해야 Claude가 올바른 코드를 생성합니다.

Step 5 · 오류 전달

오류 발생 시 Claude에게 전달하는 방법

SAC 등록 후 오류가 발생하면 당황하지 않아도 됩니다. 오류 메시지 전체를 Claude에게 전달하면 됩니다. 이때 컨텍스트를 함께 제공할수록 해결이 빠릅니다.

✓ 좋은 오류 전달 예시 "SAC에 등록하고 스토리에 배치했는데 다음 오류가 발생했습니다: Failed to construct HTMLElement
오류는 위젯을 캔버스에 드래그했을 때 바로 발생했고, 위젯이 화면에 표시되지 않습니다. 현재 widget.json과 main.js 코드를 첨부합니다."
✗ 나쁜 오류 전달 예시 "안 됩니다. 오류가 떠요."
→ 오류 메시지, 발생 시점, 현재 코드 없이 전달 시 Claude가 원인을 추정해야 하므로 해결 사이클이 늘어납니다.

오류 전달 체크리스트

  • Chrome 개발자 도구(F12) → Console 탭의 오류 메시지 전체 복사
  • 오류가 발생한 시점 명시 (등록 시, 스토리 배치 시, 버튼 클릭 시 등)
  • 현재 widget.json / main.js / styling.js 코드 전체 첨부
  • 직전에 변경한 내용이 있다면 무엇을 변경했는지 기술
실습 예제 A

메뉴 내비게이션 위젯

SAC 스토리 안에 계층형 메뉴 트리를 표시하고, 메뉴 항목 클릭 시 다른 스토리나 기능으로 이동하는 위젯입니다. 메뉴 구조는 위젯 코드에 하드코딩되어 있으며, Public Dimension 연계를 통한 동적 구성은 후속 교육에서 다룹니다.

구현된 기능

완성본 기준

2단계 계층 메뉴 트리 렌더링, 섹션 펼침/접힘(accordion), 메뉴 전체 숨기기/보이기(flip), 항목 클릭 시 SAC NavigationUtils 연동

구성 파일

widget.json + main.js

이 위젯은 외부 설정값이 없어 styling.js가 불필요합니다. 2파일로 구성됩니다.

Step 1 · 요구사항 정의

Claude에게 전달한 초기 프롬프트

다음은 메뉴 위젯 개발 시 Claude에게 전달한 정제된 모범 프롬프트입니다.

Initial Prompt — Menu Navigation Widget
SAP Analytics Cloud(SAC)에서 사용할 Custom Widget을 만들어 주세요. 이 위젯은 사업계획 패키지의 내비게이션 메뉴 역할을 합니다. [ 기능 요구사항 ] 1. 2단계 계층 메뉴를 트리 형태로 표시합니다 - 1단계: 대분류 섹션 (클릭 시 펼침/접힘) - 2단계: 세부 메뉴 항목 (클릭 시 화면 전환) 2. 위젯 상단에 메뉴 전체를 숨기고 펼치는 flip 버튼이 있어야 합니다 3. 현재 선택된 항목이 시각적으로 강조 표시되어야 합니다 4. 메뉴 항목 클릭 시 SAC의 NavigationUtils.openApplication()을 호출합니다 [ UI 디자인 ] - 배경 흰색, 강조색 네이비(#003366), 폰트 작고 깔끔하게 - 아이콘 없이 텍스트만 사용, 그림자·입체 효과 없음 - 최소 너비 200px, 높이는 위젯 크기에 맞게 자동 조절 [ SAC 환경 제약 — 반드시 준수 ] - widget.json에서 contributions 블록 사용 금지. properties/methods/events는 최상위 레벨에 선언 - widget id에 점(.)과 하이픈(-) 사용 금지, 언더스코어(_)만 사용 - main.js에서 반드시 class extends HTMLElement 문법 사용 (prototype 방식 금지) - newInstancePrefix 필드를 widget.json에 반드시 포함 - ignoreIntegrity: true 설정 [ 파일 구성 ] widget.json, main.js 2개 파일을 각각 작성해 주세요. (Styling Panel 불필요, styling.js 제외) 메뉴 데이터는 main.js 내부에 하드코딩으로 테스트용 샘플을 넣어 주세요.
Step 2 · 완성 코드

widget.json — 핵심 구조

widget.json의 전체 코드가 아닌 핵심 구조와 주의 포인트를 설명합니다.

// widget.json 핵심 구조
{
  "id": "custom_planning_nav_menu",     // 언더스코어만 사용 ← 중요
  "version": "1.0.0",
  "name": "Planning Nav Menu",
  "newInstancePrefix": "NavMenu",       // 누락 시 등록 오류 발생 ← 필수
  "ignoreIntegrity": true,            // 개발 단계에서만 true

  // ✓ contributions 블록 없이 최상위 레벨에 직접 선언
  "webcomponents": [
    {
      "kind": "main",
      "tag": "com-planning-nav-menu",
      "url": "main.js",
      "integrity": ""
    }
  ],

  "properties": {},                     // 최상위 레벨 선언
  "methods": {
    "setMenuData": {                    // SAC 스크립트에서 호출할 메서드
      "description": "메뉴 데이터 JSON 문자열 전달",
      "parameters": [
        { "name": "menuJson", "type": "string" }  // object 아닌 string ← 중요
      ]
    }
  },
  "events": {
    "onMenuSelect": {

보안 — 개발 vs 운영

API Key 노출 위험

현재 구현은 브라우저에서 API를 직접 호출하는 개발/테스트 전용 구조입니다. API Key가 브라우저 메모리에 올라가므로 내부 테스트 환경에서만 사용하십시오. 운영 배포 시에는 SAP BTP 또는 별도 서버에 Proxy를 구축하고 API Key를 서버 환경변수로만 관리해야 합니다.

운영 Proxy 구조 개요 브라우저 → SAP BTP(or Node.js) Proxy 서버 → Anthropic API. Proxy 서버가 API Key를 보관하고, SAC 테넌트 도메인만 CORS 허용합니다. 이 구조에서는 위젯 Styling Panel에 API Key 대신 Proxy 서버 URL만 입력합니다.

확인 문제

Q1SAC Custom Widget의 3개 파일 중, SAC 스토리 편집 화면 우측 패널의 설정 입력 폼을 담당하는 파일은?
styling.js는 SAC 스토리 편집 화면의 Styling Panel을 담당합니다. API Key, System Prompt 등 위젯 설정값을 입력받는 UI를 이 파일에서 구현합니다. 이 파일이 누락되면 Styling Panel에 설정 입력란이 표시되지 않습니다.
Q2widget.json의 id 필드에 사용할 수 없는 문자는?
SAC는 widget id에 하이픈(-)과 점(.)을 허용하지 않습니다. 언더스코어(_)만 사용해야 합니다. 예를 들어 my.widget-v1은 불가하며, my_widget_v1로 작성해야 합니다.
Q3SAC 스크립트에서 사용할 수 없는 구문은?
SAC 스크립트에서는 JSON.stringify(), [] 배열 리터럴, 삼항연산자(?:), str.replace() 등을 사용할 수 없습니다. JSON 문자열이 필요한 경우 + 연결 연산자로 직접 조립해야 합니다. Claude에게 SAC 스크립트 작성을 요청할 때 이 제약을 미리 명시하면 오류를 줄일 수 있습니다.
Q4브라우저에서 Anthropic API를 직접 호출할 때 반드시 포함해야 하는 헤더는?
브라우저에서 직접 Anthropic API를 호출하려면 anthropic-dangerous-direct-browser-access: true 헤더가 반드시 필요합니다. 이 헤더가 없으면 CORS 정책에 의해 Failed to fetch 오류가 발생합니다. 단, 이 방식은 개발/테스트 전용이며 운영 환경에서는 Proxy 서버 구조를 사용해야 합니다.
Q5동일한 id의 위젯을 수정하여 재배포할 때 올바른 순서는?
SAC Custom Widget은 덮어쓰기를 지원하지 않습니다. 동일한 id의 위젯을 수정하려면 반드시 기존 위젯을 삭제한 후 새로 등록해야 합니다. 삭제 전에 해당 위젯을 사용 중인 스토리가 있는지 먼저 확인하십시오.
