CO 배부를 알수록 SAC Allocation을
오해하기 쉽다

SAC Allocation은 전표를 만들지 않습니다. 하지만 결과가 없는 것도 아닙니다. CO 경험자가 반드시 교정해야 할 다섯 가지 오해를 체계적으로 다룹니다.

대상: 주니어~시니어 컨설턴트 선수과목: SAP CO 기초 소요시간: 약 45분 ASPN Group 시나리오 기반
Tab 1 / 5CO 배부의 전표 기반 구조

CO 배부: 전표 기반 배부의 작동 원리

Distribution과 Assessment는 모두 실제 회계 전표를 생성합니다. 이 구조는 SAC Allocation과의 비교에서 가장 중요한 출발점입니다.

Distribution

원가요소 유지 배부

  • Original Cost Element를 그대로 유지
  • 비용의 성격(임금, 감가상각 등)을 추적 가능
  • Sender CC → Receiver CC 직접 이전
  • 전표에 원본 계정과목 그대로 기록
Assessment

이차 원가요소로 집계 배부

  • Secondary Cost Element(43xx) 사용
  • 원가 성격이 집계 후 묻힘 (Pool → 재배부)
  • Service Center 운영 원가 배부에 적합
  • 전표에 집계 계정으로 기록

Distribution vs Assessment — 핵심 차이

비교 항목 Distribution Assessment
원가요소 유형 Primary (원본 유지) Secondary (43xx 생성)
비용 성격 추적 가능 불가 (집계됨)

Sender Credit ✔ 발생 ✔ 발생
Receiver Debit ✔ 발생 ✔ 발생

⚠ CO 배부의 구조적 특성 — SAC와 비교 시 핵심
CO 배부 = 전표 생성 + Sender Credit + Receiver Debit + Audit Trail 보유
이 네 가지 특성이 CO 배부를 "회계 기록"으로 만듭니다. SAC Allocation에는 이 중 어느 것도 존재하지 않습니다.
CO 배부의 재실행 방식

재실행 = 역전표 후 재처리

잘못된 Cycle을 재실행하려면 역전표(Reversal)를 먼저 생성한 후 Cycle을 수정·재실행해야 합니다. 이미 생성된 전표는 삭제가 아니라 상쇄 전표로 처리됩니다. 이 방식은 SAC Allocation의 재실행과 근본적으로 다릅니다.

→ Tab 2로: "전표가 없다"는 것이 SAC Allocation에서 무엇을 의미하는지, 그리고 결과는 어디에 저장되는지를 다음 탭에서 확인합니다.

Tab 2 / 5SAC Allocation의 패러다임 전환

전표 없음 ≠ 결과 없음

SAC Allocation의 가장 흔한 오해는 "전표가 없으니 진짜 배부가 아니다"입니다. 전표 없이도 배부 결과는 존재하며, 저장 방식 자체가 다릅니다.

핵심 정의 — 한 문장으로
SAC Allocation = 계획/버전 데이터에 대해 계산 결과를 재구성하는 엔진이지,
회계 전표를 생성하는 기능이 아니다.
❌ 흔한 오해
"전표가 없으면 Sender 금액이 그대로 남아 있겠네요?"
→ SAC Allocation은 Sender를 차감하고 Receiver에 적립하는 방식이 아닙니다.
✔ 실제 동작
원본 데이터는 보존되고, 배부 결과는 별도 레이어로 저장됩니다.
→ Original Data와 Allocated Data가 동시에 존재합니다.

Tab 3 / 5배부 기준 설계와 차원 자유도

Cost Center가 아니어도 됩니다

CO 배부의 Sender/Receiver는 Cost Center 중심입니다. SAC Allocation은 모델에 정의된 어떤 Dimension이든 Sender 또는 Receiver로 지정할 수 있습니다. 이것이 CO 컨설턴트들이 가장 놀라는 지점입니다.

CO 배부

Cost Center 중심 구조

  • Sender = Cost Center (반드시)
  • Receiver = Cost Center, Order, WBS, …
  • 배부 기준 = Tracing Factor (수량, 금액 등)
  • Cycle 설계 필요
SAC Allocation

모든 Dimension이 Sender / Receiver

  • Sender = 모델의 모든 Dimension
  • Receiver = 모델의 모든 Dimension
  • 배부 기준 = Driver (모델 내 값 또는 외부 입력)
  • Rule Set 설계 (Cycle 불필요)

ASPN IT
Infrastructure

APAC / EMEA / Americas

Dimension Agnostic의 핵심 메시지: SAC Allocation에서 Driver는 "어떤 비율로 나눌 것인가"를 정의하고, Dimension 자유도는 "누구에게 배부할 것인가"를 정의합니다. 두 가지 모두 모델 설계 단계에서 결정되며, CO처럼 시스템 오브젝트(CC, Order)에 종속되지 않습니다.

Tab 4 / 5실행 & 재실행 메커니즘

Clear는 자동이 아니다 — Data Action 설계의 책임

SAC Allocation Step 단독 실행은 기본적으로 델타 포스팅(Delta Posting)입니다. 재실행 시 누적을 막으려면 Data Action 내에 Clear Step을 Allocation Step 앞에 명시적으로 배치해야 합니다. 이것은 시스템이 자동으로 처리하는 것이 아니라 설계자의 책임입니다.

❌ 흔한 오해 — CO 배부 경험자가 특히 주의
"SAC Allocation을 재실행하면 이전 결과가 자동으로 지워지겠지?"
→ 아닙니다. Clear Step 없이 Allocation Step만 재실행하면 이전 배부 결과에 새 결과가 추가(Delta)됩니다. 이중 배부가 발생합니다.

Write Mode 옵션 — Allocation Step 내 설정

기본 동작 (Overwrite Target 미설정)

델타 포스팅 (Delta Posting)

  • 기존 Target 값에 배부 결과를 추가
  • 재실행 시 값이 누적됨
  • 최초 1회 실행 또는 증분 배부에만 안전
  • 반복 실행 시 이중/삼중 배부 위험
Overwrite Target 활성화

Target 덮어쓰기

  • Step 실행 시점에 해당 Target 범위를 먼저 초기화
  • Step 범위 내 누적 방지
  • 단, Step 범위 외 잔여 데이터는 미삭제
  • 범위가 명확할 때 단순 재실행에 적합

권장 패턴 — Data Action 내 Fact Deletion Step 선행 배치

Step 01

Fact Deletion Step

배부 대상 버전·기간·범위의 기존 배부 결과 레코드를 명시적으로 삭제합니다. Allocation Step과 별개로 Data Action 내 첫 번째 Step으로 구성합니다.

Step 02

Allocation Step

삭제된 빈 공간에 Driver 기반으로 배부 결과를 계산하여 저장합니다. 이전 결과가 없으므로 델타 누적 없음.

Step 03

(선택) Publish

Private Version에서 작업한 경우 결과를 Public Version으로 Publish합니다. Multi Action으로 묶어 자동화 가능합니다.

Clear Step의 두 가지 효과

효과 1

멱등성(Idempotency) 보장

동일 조건으로 몇 번 실행하든 결과가 누적되지 않고 항상 동일한 최신 배부 결과 1건만 유지됩니다. What-if 반복 시뮬레이션을 안전하게 수행할 수 있는 전제 조건입니다.

효과 2

데이터 볼륨 최적화 → 퍼포먼스 향상

배부를 반복할수록 쌓이는 불필요한 레코드를 제거합니다. SAC Planning 모델의 쿼리 성능과 배부 실행 속도는 저장된 데이터 볼륨에 직접적인 영향을 받으므로, Clear Step은 퍼포먼스 설계의 일부로 취급해야 합니다.

⚠ 주의 — 버전 보호 및 Clear 범위 정의
Fact Deletion Step의 범위(버전, 기간, 계정 등)를 배부 결과 범위와 정확히 일치시켜야 합니다. 범위가 너무 넓으면 원본 계획 데이터가 삭제될 수 있습니다. What-if 시나리오를 비교 보존하려면 별도 버전(Private/Public Version)을 복사한 후 작업하십시오.
