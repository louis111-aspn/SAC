1. CDS Hierarchy 개요

CDS Hierarchy는 DEFINE HIERARCHY라는 별도 DDL 구문으로 정의합니다. 일반 CDS View가 아니라 독립적인 오브젝트 유형이며, 파일 확장자도 .ddlx를 사용합니다. Dimension View의 @ObjectModel.hierarchyAssociation과 연결되어 SAC에서 드릴다운·롤업 분석을 가능하게 합니다.

ASPN Group — 이익센터 계층 구조 예시

ASPN Profit Center Hierarchy — 3단계 구조
ASPN_ALL ASPN Group 전체 ROOT
└─ ASPN_KR Korea Region (ASPN Korea + Holdings) NODE
└─ PC_1010 영업본부 LEAF
└─ PC_1020 관리본부 LEAF
└─ PC_1030 IT본부 LEAF
└─ ASPN_EU Europe Region (ASPN Europe) NODE
└─ PC_3010 Sales Europe LEAF
└─ PC_3020 Operations Europe LEAF
SAC 드릴다운 동작

SAC에서 이익센터 Dimension을 행에 배치하면 위 계층 구조로 드릴다운이 가능합니다. ASPN_ALL → ASPN_KR / ASPN_EU → 개별 이익센터 순으로 집계가 자동으로 롤업·드릴다운됩니다. 이 동작을 가능하게 하는 것이 DEFINE HIERARCHY로 정의된 Hierarchy View입니다.

2. Parent-Child vs. Level-Based Hierarchy

CDS Hierarchy는 데이터 저장 구조에 따라 두 가지 유형으로 나뉩니다. ASPN Group 프로젝트에서 어떤 유형을 선택할지는 마스터 데이터 테이블 구조를 먼저 확인해야 합니다.

Parent-Child Hierarchy
각 노드가 부모 노드 ID를 보유
구조각 레코드에 ParentNode 필드. 깊이 제한 없음
유연성높음 — 런타임에 계층 구조 변경 가능
SAP 표준이익센터(SETHEADER/SETLEAF), GL 계정 그룹(SKA1→SKAT) 등 대부분의 SAP 마스터 데이터
CDS 구문PARENT CHILD HIERARCHY
깊이가변적 (데이터에 따라 N레벨)
ASPN 활용이익센터 계층, GL 계정 계층
Level-Based Hierarchy (개념적 분류)
레벨별 필드가 고정된 구조로 계층 표현
구조레코드에 Level1, Level2… 등 레벨 컬럼이 고정 존재
유연성낮음 — 레벨 추가 시 스키마 변경 필요
SAP 표준시간 Dimension(연→분기→월→일), 고정 조직 계층
CDS 구현as parent child hierarchy 구문에서 레벨 속성을 활용하거나 Attribute 기반으로 표현. LEVEL HIERARCHY라는 독립 키워드는 존재하지 않음
깊이고정적 (설계 시 결정)
ASPN 활용회계연도→분기→월 시간 계층 (개념적 구분)
선택 기준

마스터 데이터 테이블에 ParentNode(부모 ID) 필드가 있으면 Parent-Child 방식으로 구현합니다. SAP S/4HANA의 이익센터, GL 계정, 코스트센터는 대부분 이 구조입니다. 시간 Dimension(연→분기→월)처럼 레벨이 고정된 경우는 개념적으로 Level-Based로 분류하지만, CDS에서는 as parent child hierarchy 구문 내 레벨 속성으로 표현하거나 Attribute 기반 계층으로 구현합니다. LEVEL HIERARCHY라는 독립적인 CDS 키워드는 존재하지 않습니다.

3. DEFINE HIERARCHY 구문 상세

Parent-Child Hierarchy — ZASPN_H_ProfitCenter

ABAP CDS · DDLX ZASPN_H_ProfitCenter — Parent-Child Hierarchy 해부
1–3
@EndUserText.label: 'ASPN Profit Center Hierarchy' @Hierarchy.parentChild.recursion.handling: #FAIL @AccessControl.authorizationCheck: #NOT_REQUIRED
Hierarchy 레벨 Annotation. #FAIL: 순환 참조 발생 시 오류. #IGNORE: 순환 무시. Hierarchy 자체의 AccessControl은 소스 Dimension View에서 제어하므로 #NOT_REQUIRED가 일반적입니다.
4–6
define hierarchy ZASPN_H_ProfitCenter as parent child hierarchy ( source ZASPN_I_ProfitCenterNode // 노드 데이터 소스 View
Hierarchy 선언. as parent child hierarchy로 유형을 지정합니다. source는 계층 노드 데이터를 제공하는 CDS View — 일반 View Entity 또는 Table Function 모두 가능합니다.
7–10
child to parent association _Parent // 부모 연결 Association start where ParentProfitCenter is initial // 루트 조건 // 또는: start where IsRoot = 'X' )
핵심 구성. child to parent association: 자식→부모 방향 Association 이름. start where … is initial: 부모 ID가 비어 있는 노드를 루트(최상위)로 지정합니다.
11–18
{ node { ControllingArea, ProfitCenter, ParentProfitCenter // 부모 ID 필드 } siblings order by ProfitCenter ascending // 동일 레벨 정렬 }
node 블록. 계층 노드가 보유해야 할 필드 목록. 키 필드 + 부모 ID 필드가 필수. siblings order by: 같은 부모를 가진 노드들의 표시 순서 — SAC에서 계층 표시 순서에 직접 영향합니다.

Hierarchy 소스 View — ZASPN_I_ProfitCenterNode

DEFINE HIERARCHY의 source는 계층 노드 데이터를 제공하는 일반 CDS View입니다. SAP 표준 이익센터 계층 테이블(SETHEADER, SETLEAF)을 조합하거나, 커스텀 계층 테이블을 소스로 사용합니다.

ABAP CDS ZASPN_I_ProfitCenterNode — 계층 노드 소스 View 
// 계층 노드 View: #DIMENSION이 아닌 일반 View
@EndUserText.label: 'ASPN Profit Center Hierarchy Node'
@AccessControl.authorizationCheck: #NOT_REQUIRED

define view entity ZASPN_I_ProfitCenterNode
  as select from ZASPN_T_PCHierarchy as Node  // 커스텀 계층 테이블

  // child → parent 방향 Self-Association
  association [0..1] to ZASPN_I_ProfitCenterNode as _Parent
    on  $projection.ControllingArea    = _Parent.ControllingArea
    and $projection.ParentProfitCenter = _Parent.ProfitCenter

{
  key Node.ControllingArea,
  key Node.ProfitCenter,

      Node.ParentProfitCenter,   // 부모 ID — DEFINE HIERARCHY의 핵심
      Node.HierarchyLevel,       // 레벨 정보 (선택)
      Node.HierarchyNode,        // 노드 이름/코드
      Node.ValidityStartDate,
      Node.ValidityEndDate,

      _Parent   // DEFINE HIERARCHY에서 참조할 Association 노출
}
where Node.ValidityEndDate >= $session.system_date
Self-Association 패턴

Parent-Child Hierarchy 소스 View는 자기 자신을 참조하는 Self-Association(_Parent)을 선언합니다. 이 Association이 DEFINE HIERARCHY의 child to parent association _Parent에서 참조됩니다. Association ON 조건은 현재 노드의 ParentProfitCenter = 부모 노드의 ProfitCenter가 핵심입니다.

4. 시간 계층 구현 — Level-Based 개념의 CDS 표현

ASPN Group의 시간 분석 계층(회계연도 → 분기 → 월)은 Level-Based 개념으로 설계합니다. CDS에서 LEVEL HIERARCHY라는 독립 키워드는 존재하지 않으며, as parent child hierarchy 구문 내에서 레벨 정보를 활용하거나 레벨별 필드가 고정된 소스 View를 기반으로 구현합니다. 아래 예시는 이 개념을 표현한 구현 패턴입니다.

ABAP CDS · DDLX (개념 예시) ZASPN_H_FiscalCalendar — 시간 계층 (레벨 고정 구조 기반) 
@EndUserText.label: 'ASPN Fiscal Calendar Hierarchy'
@Analytics.hierarchyRootNode: '0000'   // 가상 루트 노드 값

// 레벨 고정 구조: FiscalYear → FiscalQuarter → FiscalPeriod
// 소스 View의 각 레코드에 세 레벨 필드가 모두 존재
define hierarchy ZASPN_H_FiscalCalendar
  as parent child hierarchy (
    source ZASPN_I_FiscalCalendarNode

    // 자기 참조 Association으로 레벨 간 부모-자식 관계 표현
    child to parent association _Parent
    start where FiscalYear is not initial
        and FiscalQuarter is initial
        and FiscalPeriod is initial   // 연도 레벨만 있는 행 = 루트
  )
{
  node {
    FiscalYear,
    FiscalQuarter,
    FiscalPeriod
  }
  siblings order by FiscalPeriod ascending
}
@Analytics.hierarchyRootNode

Level-Based Hierarchy에서 @Analytics.hierarchyRootNode는 가상 루트 노드의 값을 지정합니다. SAC가 최상위 집계 레벨을 인식하는 데 사용됩니다. Parent-Child에서는 start where … is initial로 루트를 지정하므로 이 Annotation이 필요하지 않습니다.

📋 Level-Based 소스 View 구조 예시 — ZASPN_I_FiscalCalendarNode ▼
ABAP CDS시간 계층 노드 소스 View
@EndUserText.label: 'ASPN Fiscal Calendar Node'
@AccessControl.authorizationCheck: #NOT_REQUIRED

define view entity ZASPN_I_FiscalCalendarNode
  as select from T009B as Cal   // SAP 회계 기간 테이블
    inner join ZASPN_T_FiscalQuarter as Q
      on Cal.PERIV = Q.FiscalYearVariant
     and Cal.BUMON = Q.FiscalPeriod
{
  key Cal.GJAHR  as FiscalYear,
  key Q.FiscalQuarter,
  key Cal.BUMON  as FiscalPeriod
}
where Cal.PERIV = 'K4'   // ASPN 회계연도 변형

Level-Based 소스 View는 Self-Association이 필요 없습니다. 각 레코드에 모든 레벨 필드(FiscalYear, FiscalQuarter, FiscalPeriod)가 존재하기 때문입니다. 레벨 간 부모-자식 관계는 DEFINE HIERARCHY의 hierarchy levels 선언으로 자동 결정됩니다.

5. Dimension View와 Hierarchy View 연결

Tab 03에서 ZASPN_I_ProfitCenter에 _Hierarchy Association 포인트를 예약했습니다. 이제 DEFINE HIERARCHY로 생성한 ZASPN_H_ProfitCenter를 그 Association에 연결합니다. 연결 구조 전체를 아래 코드로 최종 완성합니다.

ABAP CDS ZASPN_I_ProfitCenter — Hierarchy Association 최종 연결 
@Analytics.dataCategory: #DIMENSION
@EndUserText.label: 'Profit Center'
@AccessControl.authorizationCheck: #CHECK

define view entity ZASPN_I_ProfitCenter
  as select from I_ProfitCenter as PC

  // Text Association (Tab 03에서 이미 선언)
  association [0..*] to I_ProfitCenterText as _Text
    on  $projection.ControllingArea = _Text.ControllingArea
    and $projection.ProfitCenter    = _Text.ProfitCenter
    and $projection.ValidityEndDate = _Text.ValidityEndDate

  // ★ Hierarchy Association: DEFINE HIERARCHY 오브젝트 참조
  association [0..1] to ZASPN_H_ProfitCenter as _Hierarchy
    on  $projection.ControllingArea = _Hierarchy.ControllingArea
    and $projection.ProfitCenter    = _Hierarchy.ProfitCenter

{
  @ObjectModel.text.association: '_Text'
  key PC.ControllingArea,

  @ObjectModel.text.association: '_Text'
  @Consumption.valueHelpDefinition: [{ entity: {
      name: 'ZASPN_I_ProfitCenter', element: 'ProfitCenter'
  }}]
  key PC.ProfitCenter,
  key PC.ValidityEndDate,

      PC.Segment,
      PC.CompanyCode,

      _Text,

    // ★ @ObjectModel.hierarchyAssociation — SAC가 Hierarchy 인식
    @ObjectModel.hierarchyAssociation: '_Hierarchy'
      _Hierarchy
}
where PC.ValidityEndDate >= $session.system_date

↑ association [0..1] to ZASPN_H_ProfitCenter as _Hierarchy
DDLS ZASPN_I_ProfitCenter @Analytics.dataCategory: #DIMENSION Dimension View (Tab 03)
SAC가 인식하는 Annotation

파일 유형
Hierarchy View → .ddlx  |  나머지 → .ddls
주의 — .ddlx 파일 유형

DEFINE HIERARCHY는 일반 CDS View(.ddls)와 다른 .ddlx 파일로 생성합니다. ADT에서 New → Other ABAP Repository Objects → Core Data Services → Hierarchy를 선택해야 합니다. .ddls 파일에 define hierarchy를 작성하면 활성화 오류가 발생합니다.

6. SAC에서 Hierarchy 동작 방식 및 주의사항

①

SAC Live Model Import 시 Hierarchy 자동 인식

SAC가 Query View를 Live Model로 Import할 때, @ObjectModel.hierarchyAssociation이 선언된 Dimension을 자동으로 계층형 Dimension으로 인식합니다. 별도 SAC 설정 없이 Hierarchy가 활성화됩니다.

②

드릴다운 집계는 Analytics Engine이 처리

SAC에서 계층 노드를 클릭해 드릴다운할 때, 하위 노드 집계는 SAP Analytics Engine(HANA 레이어)이 자동으로 수행합니다. CDS View에 집계 로직을 별도로 작성할 필요가 없습니다.

③

SIBLING ORDER — SAC 표시 순서 제어

DEFINE HIERARCHY의 siblings order by가 SAC에서 계층 노드 표시 순서를 결정합니다. 선언하지 않으면 순서가 불규칙합니다. ASPN Group에서는 이익센터 코드 오름차순, GL 계정 코드 오름차순으로 설정합니다.

④

Hierarchy 유효성 — 유효 기간 관리

소스 View의 WHERE 절(ValidityEndDate >= $session.system_date)로 만료 노드를 제외해야 SAC에서 올바른 계층이 표시됩니다. 유효 기간 관리를 누락하면 이미 조직이 변경된 이익센터가 계속 계층에 노출됩니다.

⑤

순환 참조(Cycle) 방지

@Hierarchy.parentChild.recursion.handling: #FAIL 설정을 권장합니다. 데이터 오류로 A의 부모가 B이고 B의 부모가 A인 순환이 발생하면 #FAIL 설정 시 즉시 오류로 노출되어 조기 감지가 가능합니다. #IGNORE는 순환을 무시하므로 데이터 품질 이슈를 숨길 수 있습니다.
